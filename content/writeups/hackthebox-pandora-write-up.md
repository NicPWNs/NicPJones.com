<p>This is my write-up for the Pandora machine on HackTheBox that just retired! Here I detail the penetration testing steps taken to scan, exploit, and privilege escalate on this target machine. This machine is categorized as <em>easy</em> difficulty and was retired on May 21, 2022.</p><h2>Pandora Summary</h2><figure><img alt="" src="/writeups/hackthebox-pandora-write-up/img-1.png"  decoding="async" width="300" height="300" loading="eager" fetchpriority="high" /></figure><h3>Target Information</h3><p><a href="https://app.hackthebox.com/machines/pandora"><strong>Machine Page</strong></a><br><strong>IP Address:</strong> 10.10.11.136<br><strong>Hostname:</strong> pandora.htb</p><h3>Synopsis</h3><p>Open SNMP with a default community string allows for enumeration which reveals SSH credentials. SSH access reveals an internal web site which can be accessed via an SSH tunnel. The Pandora FMS web application running on the internal site is vulnerable to a SQL injection vulnerability that allows for authentication bypass. The Pandora FMS web application is also vulnerable to remote code execution once authenticated which allows for a reverse shell on the target. A SUID binary does not reference the full executable path of a dependency so poisoning the path leads to privilege escalation.</p><h2>Scanning</h2><h3>Nmap (TCP)</h3><p>The Nmap scan shows that there is an Apache httpd server on port 80/tcp and SSH open on port 22/tcp.</p><pre># nmap -sV -sC -p- 10.10.11.136<br>Starting Nmap 7.92 ( https://nmap.org ) at 2022-05-20 23:18 GMT<br>Nmap scan report for 10.10.11.136<br>Host is up (0.050s latency).<br>Not shown: 65533 closed tcp ports (reset)<br>PORT   STATE SERVICE VERSION<br>22/tcp open  ssh     OpenSSH 8.2p1 Ubuntu 4ubuntu0.3 (Ubuntu Linux; protocol 2.0)<br>| ssh-hostkey: <br>|   3072 24:c2:95:a5:c3:0b:3f:f3:17:3c:68:d7:af:2b:53:38 (RSA)<br>|   256 b1:41:77:99:46:9a:6c:5d:d2:98:2f:c0:32:9a:ce:03 (ECDSA)<br>|_  256 e7:36:43:3b:a9:47:8a:19:01:58:b2:bc:89:f6:51:08 (ED25519)<br>80/tcp open  http    Apache httpd 2.4.41 ((Ubuntu))<br>|_http-title: Play | Landing<br>|_http-server-header: Apache/2.4.41 (Ubuntu)<br>Service Info: OS: Linux; CPE: cpe:/o:linux:linux_kernel</pre><pre>Service detection performed. The only place to try the credentials dumped from SNMP is SSH on port 22/tcp. The credentials work for SSH!</p><figure><img alt="Successful SSH login with found credentials." src="/writeups/hackthebox-pandora-write-up/img-3.png"  decoding="async" width="550" height="534" loading="lazy" /><figcaption>Successful SSH login with found credentials.</figcaption></figure><h3>Alternate Web Server</h3><p>This daniel user isn&#39;t yet the user that gives us the user.txt flag, so there&#39;s still more to do before privilege escalation. After looking around, the Apache httpd directory /var/www and the /etc/hosts file indicate that there is an alternate website running something called Pandora.</p><figure><img alt="/var/www and /etc/hosts suggesting other web server." src="/writeups/hackthebox-pandora-write-up/img-4.png"  decoding="async" width="517" height="193" loading="lazy" /><figcaption>/var/www and /etc/hosts suggesting other web server.</figcaption></figure><h3>Tunnels and Proxies</h3><p>In order to access the new web site on the target machine with our host’s web browser and tools, we must first establish a tunnel to the target machine, and then proxy any web browsers and tools to that established tunnel. A tunnel can be quickly set up using the SSH access we have. In a new SSH session, simply adding -D 80 creates a dynamic tunnel to port 80/tcp on our localhost.</p><pre># sudo ssh -D 80 daniel@10.10.11.136</pre><p>Now, with the tunnel set up to port 80/tcp on our localhost, we must instruct our browser to proxy to that tunnel. This can be done in the browser directly or through an easy to use tool like <a href="https://addons.mozilla.org/en-US/firefox/addon/foxyproxy-standard/">FoxyProxy</a>.</p><figure><img alt="Using FoxyProxy to set up the SOCKS5 proxy to the localhost tunnel on port 80/tcp." src="/writeups/hackthebox-pandora-write-up/img-5.png"  decoding="async" width="1024" height="440" loading="lazy" /><figcaption>Using FoxyProxy to set up the SOCKS5 proxy to the localhost tunnel on port 80/tcp.</figcaption></figure><p>Now we can see the Pandora web site at <a href="http://127.0.0.1/pandora_console/.">http://127.0.0.1/pandora_console/.</a></p><figure><img alt="Landing page for Pandora FMS." src="/writeups/hackthebox-pandora-write-up/img-6.png"  decoding="async" width="1024" height="632" loading="lazy" /><figcaption>Landing page for Pandora FMS.</figcaption></figure><h3>CVE-2021–32099</h3><p>After searching around about this software Pandora FMS there is a SQL injection vulnerability registered as CVE-2021–32099 affecting this version v7.0NG.742. I found this <a href="https://github.com/ibnuuby/CVE-2021-32099">GitHub repo</a> which provides a link to automatically set our PHPSESSID cookie to one that is extracted via the SQL injection vulnerability. The link that we use is below.</p><pre><a href="http://127.0.0.1:80/pandora_console/include/chart_generator.php?session_id=a%27%20UNION%20SELECT%20%27a%27,1,%27id_usuario|s:5:%22admin%22;%27%20as%20data%20FROM%20tsessions_php%20WHERE%20%271%27=%271">http://127.0.0.1:80/pandora_console/include/chart_generator.php?session_id=a%27%20UNION%20SELECT%20%27a%27,1,%27id_usuario|s:5:%22admin%22;%27%20as%20data%20FROM%20tsessions_php%20WHERE%20%271%27=%271</a></pre><p>Going to this link sets our PHPSESSID cookie to a valid administrator one. So when we go back to the landing page we are logged in as the admin without needing a username or password.</p><figure><img alt="Logged in as admin on Pandora FMS." src="/writeups/hackthebox-pandora-write-up/img-7.png"  decoding="async" width="1024" height="629" loading="lazy" /><figcaption>Logged in as admin on Pandora FMS.</figcaption></figure><h3>CVE-2020–5844</h3><p>Now, with admin login we still need some form of code execution on the system to move further. Searching again shows there is an authenticated remote code execution vulnerability registered as CVE-2020–5844 affecting this version of Pandora FMS v7.0NG.742. There are a few proof of concept scripts available for exploiting this vulnerability, but I have since created my own <a href="https://github.com/UNICORDev/exploit-CVE-2020-5844">exploit for CVE-2020–5844</a> under my <a href="https://unicord.dev/">UNICORD</a> project.</p><p>My exploit script can be used in multiple different configurations including web shell, custom command, and reverse shell. For our purposes, we’ll use the reverse shell functionality and provide the PHPSESSID stored in our browser that was stolen earlier. We also must use proxychains in order to route the traffic to the established SSH tunnel. This can be done by adding the line socks5 127.0.0.1 80 to the end of /etc/proxychains.conf. Using the proxychains -q command at the beginning of the exploit command routes the traffic through the proxy.</p><figure><img alt="Successful exploitation of CVE-2020–5844 for a reverse shell." src="/writeups/hackthebox-pandora-write-up/img-8.png"  decoding="async" width="939" height="362" loading="lazy" /><figcaption>Successful exploitation of CVE-2020–5844 for a reverse shell.</figcaption></figure><p>Now, although the reverse shell works pretty good, SSH access is always preferred. We can add an SSH public key to this new user’s configuration to log in with SSH.</p><figure><img alt="Creating SSH config with correct permissions." src="/writeups/hackthebox-pandora-write-up/img-9.png"  decoding="async" width="353" height="66" loading="lazy" /><figcaption>Creating SSH config with correct permissions.</figcaption></figure><figure><img alt="Adding SSH public key to authorized_keys for the user Matt." src="/writeups/hackthebox-pandora-write-up/img-10.png"  decoding="async" width="1024" height="54" loading="lazy" /><figcaption>Adding SSH public key to authorized_keys for the user Matt.</figcaption></figure><p>Matt also has user.txt, so that&#39;s one flag down!</p><h2>Enumeration</h2><h3>SUID Binaries</h3><p>One of the first things I always check when enumerating for privilege escalation is SUID binaries. We can check the entire system for SUID binaries with the command below. There’s one particular SUID binary that stands out at /usr/bin/pandora_backup.</p><pre>$ find / -perm -u=s -type f 2&gt;/dev/null<br>/usr/bin/sudo<br>/usr/bin/pkexec<br>/usr/bin/chfn<br>/usr/bin/newgrp<br>/usr/bin/gpasswd<br>/usr/bin/umount<br>/usr/bin/pandora_backup<br>/usr/bin/passwd<br>/usr/bin/mount<br>/usr/bin/su<br>/usr/bin/at<br>/usr/bin/fusermount<br>/usr/bin/chsh</pre><h2>Root</h2><p>We can run this binary in the context of the root user, but we need to know what it does in order to exploit it. I copied the compiled binary to my system and ran strings on it. This indicated that the binary was running the tar command but did not explicitly specify to use /usr/bin/tar. This means that the system will automatically search in our PATH for a tar executable. We can edit our PATH, so this means we can create a malicious tar executable and add its location to the beginning of our PATH. This way, pandora_backup will end up running our fake tar with root privileges rather than the real one. We can create a basic fake tar and edit our PATH to include it like below. Running the SUID binary gives root!</p><figure><img alt="Creating a malicious tar executable and editing the PATH to achieve root." src="/writeups/hackthebox-pandora-write-up/img-11.png"  decoding="async" width="371" height="162" loading="lazy" /><figcaption>Creating a malicious tar executable and editing the PATH to achieve root.</figcaption></figure><p>That’s root! The root.txt flag is then available in /root.</p><h2>Loot</h2><p>Other than the points on HackTheBox, the lessons learned are the real treasures for this box.</p><ol><li>Missing Absolute Paths — A script or binary not referencing the absolute path such as /usr/bin/tar for one of its dependencies is an easy mistake and a crucial one in the case of an SUID binary.</li><li>Creating Exploits — Creating your own exploits and tools is often a great way to learn a lot about a particular vulnerability while also practicing coding along the way.
